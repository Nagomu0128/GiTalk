# 02 - コア機能要件（Git的機能）

## 概要

会話ツリーに対して Git ライクな操作を GUI で提供する。
すべての操作はターミナルではなく、ビジュアルなUI上で行う。

## 基本操作

### チャット（会話の基本フロー）

1. ユーザーが新しい会話（Conversation）を作成する
2. デフォルトブランチ "main" が自動作成される（`active_branch_id` = main）
3. **空の会話状態:** ノードが0個の状態ではツリービューは非表示。チャットビューにウェルカム画面（「メッセージを送信して会話を始めましょう」）を表示。ブランチ作成・Git操作は無効
4. ユーザーがメッセージを送信 → Gemini API にコンテキスト付きで送信 → 応答を受け取る
5. リクエスト＋レスポンスのペアが1ノードとしてツリーに追加される（`branch_id` = アクティブブランチ）
6. ブランチの `head_node_id` が新ノードに更新される（楽観的ロック: `WHERE head_node_id = :expected`。競合時は `409 CONFLICT`）
7. **初回メッセージ後:** ルートノード作成と同時にツリービューが出現（アニメーション）。バックグラウンドでタイトル自動生成（Gemini に「この会話に短いタイトルをつけてください」と送信、非同期で Conversation.title を更新）

## Git的操作の定義

### branch（ブランチ作成）

**操作:** 任意のノードから新しいブランチを作成する。

- **遡及的分岐:** 最新ノードだけでなく、過去の任意のノードから分岐可能
- ブランチ名はユーザーが指定（デフォルトで自動命名も可）
- 作成時、`base_node_id` = 分岐元ノード、`head_node_id` = 分岐元ノード（まだ会話がないため同一）

```
操作前:
[N1] → [N2] → [N3]  (main)

操作: N2 から "idea-a" ブランチを作成

操作後:
[N1] → [N2] → [N3]  (main)
              ↘
               (idea-a, head=N2, base=N2)
```

**UI:** ノードを右クリック or ノード上のアクションメニューから「ブランチを作成」

### switch / checkout（ブランチ切替）

**操作:** アクティブなブランチを切り替える。

- 切替後、そのブランチの `head_node_id` のノードが表示される
- 新しい会話はそのブランチの head に追加される
- ツリーの可視化上で、アクティブブランチがハイライトされる

**UI:** ブランチ一覧パネルから選択 / ツリー上のブランチラベルをクリック

### merge（ブランチ統合）

**操作:** マージ元ブランチの内容を要約し、マージ先ブランチの先端に要約ノードを追加する。

**手順:**
1. ユーザーがマージ元とマージ先を指定
2. マージ元ブランチの `base_node_id` から `head_node_id` までのパス上のノードを取得
3. AIにそのノード群を要約させる（要約の粒度はユーザー選択、デフォルトは「詳細」）
4. 要約ノード（`node_type: summary`）をマージ先の head に追加
5. 要約ノードの `metadata` にマージ元情報を記録
6. マージ元ブランチはそのまま残る（削除しない）。マージ後も会話を続けられる

**要約戦略とプロンプト:**

| 戦略 | 説明 | 目標トークン数 |
|------|------|--------------|
| `concise` | 1-2文で要点のみ | ~200 tokens |
| `detailed`（デフォルト） | 主要な論点をすべて含む | ~500 tokens |
| `conclusion_only` | 最終的な結論だけ | ~100 tokens |

要約プロンプト（Gemini API に送信）:
```
以下は「{ブランチ名}」というトピックでの一連の会話です。
この会話を{戦略に応じた指示}で要約してください。

{戦略別の指示}
- concise: 「1-2文で要点のみを簡潔にまとめてください」
- detailed: 「主要な論点、議論の流れ、得られた結論をすべて含めて要約してください」
- conclusion_only: 「最終的な結論・決定事項のみを抽出してください」

会話内容:
{ノード群の user_message と ai_response を時系列順に連結}
```

**同一ブランチの二重マージ:** 同じマージ元ブランチを再度マージすることは可能。前回のマージ以降に新しい会話が追加されている場合に有用。

```
操作前:
[N1] → [N2] → [N3]           (main, head=N3)
              ↘
               [N4] → [N5]   (idea-a, head=N5)

操作: idea-a を main に merge

操作後:
[N1] → [N2] → [N3] → [N6: idea-aの要約]  (main, head=N6)
              ↘
               [N4] → [N5]                 (idea-a, そのまま残る)
```

**要約ノードの metadata:**
```json
{
  "merge_source_branch_id": "<idea-aのID>",
  "merge_source_head_node_id": "<N5のID>",
  "summary_strategy": "detailed"
}
```

**UI:** ブランチ一覧から「マージ」ボタン / ドラッグ&ドロップでブランチを統合

### reset（リセット）

**操作:** ブランチの head を過去のノードに戻す。

- `head_node_id` を指定ノードに更新する
- reset で飛ばされたノードは削除せず、ツリー上に**半透明で表示**される（どのブランチの head からも到達不能な状態）
- ユーザーは孤立ノードから新しいブランチを作成して復帰させることができる
- 新しい会話は reset 先のノードから続く

```
操作前:
[N1] → [N2] → [N3] → [N4]  (main, head=N4)

操作: main を N2 に reset

操作後:
[N1] → [N2]  (main, head=N2)
         ↘
          [N3] → [N4]  (孤立ノード、ツリー上には表示)
```

**UI:** ノードを右クリック →「ここまで戻す」 / タイムライン上でスライダー操作

### cherry-pick（ノード取り込み）

**操作:** 別ブランチの特定ノードの内容を、現在のブランチに取り込む。

- 選択したノードの `user_message` と `ai_response` をコピーして新ノードを作成
- 新ノードの `parent_id` は現在のブランチの head
- `model` は元ノードの値をコピー（再生成はしない）
- `token_count` は元ノードの値をコピー
- `metadata` にcherry-pick元のノードIDを記録

```json
{
  "cherry_picked_from": "<元ノードID>"
}
```

**UI:** 別ブランチのノードを右クリック →「このノードを取り込む」

### diff（差分表示）

**操作:** 二つのブランチの分岐後の会話内容を比較表示する。

- 共通祖先（LCA: Lowest Common Ancestor）を特定
- 各ブランチの共通祖先以降のノードを並べて表示
- テキストの差分ではなく、ノード単位の比較

**UI:** 左右分割ビューで二つのブランチの会話を並列表示

### clone（クローン）

**操作:** Repository からローカル（Conversation）にコピーする。

- Repository のスナップショットを元に新しい Conversation を作成
- 全ブランチ・全ノードを複製
- clone 元の Repository への参照を保持

**UI:** リポジトリ詳細画面から「クローン」ボタン

### ブランチ削除

**操作:** ブランチを削除する。

- デフォルトブランチ（`is_default = TRUE`）は削除不可
- ブランチを削除しても、そのブランチに属するノードは削除しない（孤立ノードとしてツリーに残る）
- 削除前に確認ダイアログを表示

**UI:** ブランチ一覧のコンテキストメニューから「ブランチを削除」

### ストリーミング中断時の挙動

- Gemini API からのストリーミングが中断された場合（ネットワークエラー、タイムアウト等）、**ノードは保存しない**
- 部分的な応答はフロントエンドに表示されるが、DB には永続化されない
- ユーザーにエラーメッセージを表示し、再送信を促す
- ブランチの `head_node_id` は変更されない

### DB保存失敗時の挙動

- Gemini API のストリーミングは正常完了したが、Node の DB 保存が失敗した場合:
  1. SSE で `type: "save_failed"` イベントを送信
  2. フロントエンドは応答テキストをメモリに保持し、「保存に失敗しました。再試行しますか？」ボタンを表示
  3. ユーザーが再試行 → `POST /v1/conversations/:id/retry-save` にテキスト + メタデータを送信して保存のみ実行
  4. ブランチの `head_node_id` は保存成功まで変更されない

### タイトル自動生成

- 初回メッセージの応答完了後、バックグラウンドで非同期にタイトルを生成
- Gemini API に「以下の会話に20文字以内の短いタイトルをつけてください」と送信
- 生成されたタイトルで `Conversation.title` を UPDATE
- ユーザーは生成されたタイトルを後から手動で変更可能
- タイトル生成の失敗はユーザーに通知しない（デフォルトタイトル「新しい会話」のまま）

## 操作の一覧と優先度

| 操作 | 優先度 | 複雑度 | MVP必須 |
|------|--------|--------|---------|
| branch | 高 | 低 | Yes |
| switch/checkout | 高 | 低 | Yes |
| merge | 中 | 高 | Yes |
| reset | 中 | 低 | Yes |
| cherry-pick | 低 | 中 | No（MVP後） |
| diff | 中 | 中 | Yes |
| clone | 低 | 中 | No（MVP後） |
